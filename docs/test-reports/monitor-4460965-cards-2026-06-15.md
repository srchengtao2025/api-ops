## health_24h 字段定义 (替代旧 latest_health)

```go
// internal/dal/ops_repo.go Channel24hSummary struct
type Channel24hSummary struct {
    ChannelID       int     `json:"channel_id"`
    RequestCount    int64   `json:"request_count"`     // SUM 24h 5min 桶
    ErrorCount      int64   `json:"error_count"`       // SUM 24h
    ErrorRate       float64 `json:"error_rate"`        // SUM(err)/SUM(req) 重算
    P95LatencyMs    int     `json:"p95_latency_ms"`    // MAX(5min 桶跨 24h, 等于最新一桶值)
}
```

**为什么不用 latest 桶**:
- latest 是最新 5min, 跨 24h 不准 (漏高峰)
- SUM 24h = 实际 24h 错误率
- P95 走 channel_health_5min MAX(跨 24h) = 最新桶 (cache 已算好)

## 数据准确性自检 (ch 110)

### SQL 直查 (RoDB)

```sql
SELECT
  channel_id,
  COUNT(*) AS req,
  COUNT(*) FILTER (WHERE type = 5) AS err,
  COUNT(*) FILTER (WHERE type = 5)::numeric / COUNT(*)::numeric AS rate
FROM logs
WHERE created_at >= NOW() - INTERVAL '24 hours' AND channel_id = 110
GROUP BY channel_id;
```

**结果**: req=112, err=54, rate=0.4821

## 关键经验

1. **SUM 跨桶 比 latest 单桶准** — 跨 24h 错误率才有业务意义, latest 只是当前 5min 切片
2. **HAVING 排除无调用渠道** — 比 Go 端 filter 省内存 (DB 端聚合 + 过滤)
3. **auto-fill grid** — 替代表格在小屏更直观, 360px minmax 跨屏自适应
4. **cache 桶 MAX(P95) 跨 24h** — 等于最新桶值 (P95 单调性), 不用走 percentile_cont
