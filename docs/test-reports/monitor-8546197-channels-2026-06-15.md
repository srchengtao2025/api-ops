## 基础设施修复 (P1): main.go 漏启 scheduler.Run

### 现象

监控中心首版部署后, `/api/monitor/channels` 返 44 渠道但 `latest_health` 字段**全 null**. channel_health_5min 表 0 行.

### 根因

`cmd/server/main.go` 顶部 import 了 monitor 包, 但**漏启** scheduler.Run. monitor tick 永远不跑, 5min 桶永远不写.

### 验证 (修复前)

```bash
docker logs api-ops-api --since 1m
# (没看到任何 [monitor] tick 行)
```

### 修复 (本 PR)

`cmd/server/main.go` (line 21-25, 137-138):

```go
// 顶部 import scheduler
import (
    "github.com/api-ops/api-ops/internal/monitor"
    "github.com/api-ops/api-ops/internal/scheduler"
)

// AI scheduler 前面启 monitor scheduler
scheduler.Run(rootCtx, cfg)
log.Println("[main] P1 monitor scheduler started (1min tick: 5min aggregate + alert eval)")
```

### 验证 (修复后)

```bash
docker logs api-ops-api --since 1m
# [main] P1 monitor scheduler started (1min tick: 5min aggregate + alert eval)
# [monitor] tick: 5min_buckets=6 1h_buckets=0 alerts_fired=0
```

启动 5s 后 tick 跑, 5min_buckets=6 (1h 后 49 渠道都该有数据).

### 推广 (AGENTS.md 部署铁律 #15)

任何新加的后台 tick 引擎 (cron / sync / aggregate / monitor) **必须**走 scheduler, 不要自己 `time.Ticker` 散在 main.go.

## 关键经验

1. **漏启 scheduler 是高频 bug** — 加新 tick 引擎必走 scheduler, 不要 time.Ticker 散在 main.go
2. **tick 数据延迟** — 5min 桶 1min 写 1 次, 启动后 5s 才有第 1 个桶, 1h 后 49 渠道满
3. **告警模块暂缓** — 用户决策: rule/alert/ack/resolve/飞书推送**前端不开放**, API 就绪 (注释标记)
4. **SVG 双轴线 vs echarts** — 渠道健康这种小数据量 SVG 够用, 60% 体积优势
